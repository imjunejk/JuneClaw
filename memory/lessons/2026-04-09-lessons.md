# Daily Lessons — 2026-04-09

## Problems

1. **AgiTQ 실행 시간 3번 연속 수정** — 13:05→12:55→12:57. 최적 시간을 사전 분석 없이 반복 변경.
2. **SPLG→SPYM→SPY 티커 3단계 변경** — 하루 안에 기초 ETF를 3번 바꿈. SPLG 제거 후 SPYM 통합, 다시 SPY로 유동성 이유.
3. **SEPA Manager 육사 전략 미인식** — 신규 전략(육사) 추가 시 기존 SEPA Manager가 인식 못함. SGOV/AgiTQ 포지션을 매도 대상으로 잘못 분류.

5. **Tuner override 재발** — 이전에 차단한 MIN_TT_SCORE/STOP_LOSS_PCT 튜너 오버라이드가 다시 풀림.
6. **quick-responder max-turns 에러** — JuneClaw에서 max-turns=3이 너무 작아 "Reached max turns" 에러 발생.

## Root Causes

1. **사전 분석 부족 (AgiTQ 시간, 티커 선정)** — 실행 시간과 티커를 결정하기 전에 유동성, 스프레드, 체결률 데이터를 먼저 분석하지 않음. 감으로 설정 후 수정 반복.
2. **신규 전략-기존 시스템 통합 테스트 부재** — 육사 전략을 추가하면서 SEPA Manager와의 상호작용을 사전 검증하지 않음. Manager가 어떤 포지션을 관리하는지 범위 정의가 불명확.
3. **차단 설정의 영속성 미보장** — tuner override 차단이 코드 변경 시 실수로 풀리는 구조. 차단이 config가 아닌 코드 로직에 하드코딩되어 있어 리팩토링 시 유실.
4. **JuneClaw 기본값 보수적 설정** — max-turns 기본값을 너무 낮게(3) 잡아서 실사용 시 즉시 한계 도달.

## Solutions Applied

1. AgiTQ 실행 시간 최종 12:57 + 미체결 시 13:02 애프터장 limit order 추가
2. SPYM→SPY로 최종 확정 (유동성 우위)
3. SEPA Manager에 SGOV/AgiTQ 제외 로직 + 자동매도 방지 추가
4. 자동매매 충돌 3건 개별 해결
5. MIN_TT_SCORE/STOP_LOSS_PCT tuner override 재차단
6. quick-responder max-turns 3→5로 상향

4. **tuner override**: 차단 규칙을 config 파일로 분리하여 코드 리팩토링에도 유지되도록 설계

## Pattern Detection

1. **반복 수정 패턴 (Severity: HIGH)** — AgiTQ 실행 시간 3회, 티커 3단계 변경. 데이터 기반 의사결정 없이 직감으로 설정 후 수정 반복. 4/8에도 전략 할당 비율 3번 변경(Rule #2)과 동일 패턴.
2. **신규 전략 통합 충돌 패턴 (Severity: HIGH)** — 새 전략 추가 시 기존 시스템과 충돌. 4/8 갭 필터 방향 분리(Rule #3)와 유사하게, 새로운 요소가 기존 로직과 어떻게 상호작용하는지 사전 검증 부족.
3. **차단 설정 유실 패턴 (Severity: MEDIUM)** — 한번 수정한 것이 후속 리팩토링에서 원복되는 문제. 설정과 코드 분리 미흡.
4. **보수적 기본값 패턴 (Severity: LOW)** — 개발 시 설정한 보수적 기본값이 운영에서 문제 발생. 환경별 config 분리 필요.
